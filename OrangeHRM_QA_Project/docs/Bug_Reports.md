## Баг-репорты

### **BUG-001: Календарь позволяет выбрать дату в прошлом при создании заявки на отпуск**

**Статус:** Новый  
**Приоритет:** High  
**Серьёзность:** Major  
**Тип:** Functional/Validation  

**Окружение:** 
- Браузер: Mozilla Firefox
- ОС: Windows 11
- URL: https://opensource-demo.orangehrmlive.com/web/index.php/leave/viewLeaveList

**Описание:**  
При создании заявки на отпуск (Apply Leave) календарь не блокирует выбор дат в прошлом. Пользователь может выбрать любую дату, включая вчерашний день или более ранние даты, что противоречит бизнес-логике системы отпусков.

**Шаги воспроизведения:**
1. Авторизоваться как `Admin` / `admin123`
2. Перейти в меню Leave → Apply Leave
3. Нажать на поле "From Date" для открытия календаря
4. В календаре переключиться на предыдущий месяц
5. Выбрать любую дату в прошлом (например, вчера)
6. Заполнить остальные поля формы корректными данными
7. Нажать кнопку "Submit"

**Ожидаемый результат:**  
- Календарь не позволяет выбирать даты в прошлом (disabled state для прошедших дат) ИЛИ
- При попытке отправить форму с датой в прошлом появляется ошибка: "From Date cannot be in the past"

**Фактический результат:**  
- Календарь позволяет выбрать любую дату в прошлом
- Форма отправляется успешно без ошибок
- Заявка создаётся с некорректной датой

**Скриншот:**  
screenshots/BUG-001_past_date_calendar.png  
screenshots/BUG-001_form_submitted_with_past_date.png

**Комментарии:**  
Баг воспроизводится стабильно. Критично для бизнес-логики: сотрудники могут задним числом оформлять отпуска, что нарушает учёт рабочего времени.

### **BUG-002: Сортировка таблицы сотрудников не работает при активном фильтре поиска**

**Статус:** Новый  
**Приоритет:** Low  
**Серьёзность:** Trivial  
**Тип:** Functional  

**Окружение:** 
- Браузер: Mozilla Firefox
- ОС: Windows 11
- URL: https://opensource-demo.orangehrmlive.com/web/index.php/pim/viewEmployeeList

**Описание:**  
При применении фильтра поиска сотрудников (по имени) сортировка по колонкам таблицы (Name, Employee ID, Status) не работает корректно. Клик по заголовку колонки меняет визуальный индикатор сортировки (стрелку), но порядок записей не изменяется.

**Шаги воспроизведения:**
1. Авторизоваться и перейти в PIM → Employee List
2. В поле "Employee Name" ввести: `a` (или любую букву для получения нескольких результатов)
3. Нажать кнопку "Search"
4. Дождаться отображения отфильтрованных результатов
5. Нажать на заголовок колонки "Name" для сортировки
6. Нажать повторно для обратной сортировки

**Ожидаемый результат:**  
Записи сортируются по алфавиту (A–Z при первом клике, Z–A при повторном)

**Фактический результат:**  
- Визуальный индикатор сортировки (стрелка) меняется
- Порядок записей в таблице не изменяется
- Сортировка работает только без активного фильтра

**Скриншот:**  
screenshots/BUG-003_sort_not_working_with_filter.png

**Комментарии:**  
Воспроизводится стабильно при любом активном фильтре. Не критично для основной функциональности, но ухудшает UX при работе с большими списками сотрудников.
